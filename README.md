## 🧠 My Problem-Solving Framework

When approaching any algorithmic problem, I follow a strict 4-stage engineering process rather than just jumping into code:
1. **Analyze Constraints:** Extract bounds to determine the target Time & Space complexity ($O(N)$, $O(\log N)$, etc.).
2. **Handle Edge Cases:** Guard clauses for empty inputs, bounds overflow (`int` vs `long`), and structural anomalies.
3. **Draft the Brute-Force Solution:** Mentally map out the naive approach ($O(N^2)$) to establish a baseline.
4. **Optimize using Data Structures:** Leverage the optimal tool from my toolkit (`HashSet`, `Dictionary`, `Stack`, etc.) to hit peak execution speed.

## 🧰 The DSA Engineer's Toolkit

Here is how I visually map out the tools I use to break down complex logic:

* **`HashSet<T>`** ➡️ For Instant Lookups $O(1)$ and strict uniqueness.
* **`Dictionary<TKey, TValue>`** ➡️ To track frequencies, indices, and key-value relations.
* **`Stack<T>` / `Queue<T>`** ➡️ For dynamic tracking (LIFO / FIFO linear processes).
* **`StringBuilder`** ➡️ Used heavily for optimized mutable string manipulations to avoid memory fragmentation ($O(1)$ updates).
